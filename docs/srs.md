# <a name="head1"></a> 1. Εισαγωγή
## <a name="head11"></a> 1.1 Σκοπός  
Το παρακάτω κείμενο αποτελεί περιγραφή για τις προδιαγραφές απαιτήσεων λογισμικού. Θα αναλυθούν κυρίως σε πρωτόλειο στάδιο τα χαρακτηριστικά της εφαρμογής και οι τρόποι υλοποίησης.
## <a name="head12"></a> 1.2 Συμβάσεις Σύνταξης Έγγραφου
Το παρόν κείμενο έχει υλοποιηθεί σε MarkDown. Κάθε μέρος του κειμένου έχει χωριστεί σε κατηγορίες και υποκατηγορίες.
## <a name="head13"></a> 1.3 Προβλεπόμενο κοινό και προτάσεις ανάγνωσης 
Το κείμενο έχει δημιουργηθεί από φοιτητές και πρόκειται για μια εφαρμογή στα πλαίσια του μαθήματος Τεχνολογία Λογισμικού. Το κοινό στο οποίο απευθύνεται η εφαρμογή , πέρα από τους φοιτητές για τον εκπαιδευτικό σκοπό του μαθήματος, είναι εργαζόμενοι στον κλάδο του προγραμματισμού, οι οποίοι λειτουργούν στις ομάδες τους με τον κύκλο της μεθόδου ανάπτυξης λογισμικού Agile και πιο συγκεκριμένα με την τεχνική Scrum.
## <a name="head14"></a> 1.4 Περίληψη Εφαρμογής
Η εφαρμογή επιτρέπει την επικοινωνία μιας προγραμματιστικές ομάδας με την τεχνική Scrum. Δηλαδή, είναι μια εφαρμογή που χρησιμοποιείται για την καλύτερη οργάνωση ομάδας προγραμματιστών, με σκοπό να μπορέσουν να εκτελέσουν συγκεκριμένα μέρη ενός μεγαλύτερου προγράμματος. Για κάθε project, που δημιουργεί ένας χρήστης, μπορεί να προσκαλέσει τους υπόλοιπους προγραμματιστές και να δημιουργήσει κάποιο issue για το project. 

Τα issues αποτελούνται από 3 μέρη, τα epics και τα stories και τα tasks/bugs. Το epic πρόκειται για ένα κεντρικό σκοπό της υλοποίησης της εφαρμογής που διαιρείται σε stories. Ενώ το story διαιρείται σε διάφορα tasks έτσι ώστε να υλοποιηθεί κάποια λειτουργία του epic. Ακόμη υπάρχουν και τα bugs όπου πρόκειται για την επίλυση προγραμματιστικών λαθών της εφαρμογής που χρήζουν διόρθωση. Τέλος, η ομάδα καλείται να φέρει εις πέρας κάποια tasks, μέσα σε κάποιο χρονικό διάστημα, γνωστό και ως sprint. Όταν τελειώσει ένα sprint, ξεκινάει το επόμενο, με καινούργια tasks.
## <a name="head15"></a> 1.5 Βιβλιογραφία 
Δεν έχει χρησιμοποιηθεί κάποια βιβλιογραφία ακόμη.
# <a name="head2"></a> 2. Γενική Περιγραφή 
## <a name="head21"></a> 2.1 Λειτουργικότητα Εφαρμογής
Παρακάτω περιγράφονται πιθάνα σενάρια της Εφαρμογής
### 2.1.1 Starting User's Scenarios
`Login Scenario`: Ο χρήστης μπορεί να συνδέεται στον λογαριασμό του βάση του email/username και password που έχει δώσει.<br>
`Register Scenario`: O χρήστης μπορεί να δημιουργήσει καινούργιο λογαριασμό με μοναδικό email και username. <br>
`Payment Plan Scenario`: Ο χρήστης μπορεί να επιλέξει το συνδρομητικό πλάνο για την εφαρμογή, που καλύπτει τις ανάγκες του. <br>
`HomePage Scenario`: Ο χρήστης θα μπορεί να πλοηγηθεί στα Projects του, στο Backlog του, στις Ρυθμίσεις του λογαριασμού του, και να αποσυνδεθεί. 
### 2.1.2 Projects Scenarios
`Project Scenario`: Ο χρήστης θα μπορεί να ελέγξει όλα τα projects που συμμετέχει. Να τα αναζητήσει, να τα αρχειοθετήσει, να ελέγξει την πορεία τους, να αλλάξει τα δεδομένα τους, και να δει περαιτέρω πληροφορίες για αυτά. <br>
`Sprints Scenario`: Ο χρήστης μπορεί να δει σε τη κατάσταση βρίσκεται ένα sprint και να το επεξεργαστεί (π.χ. να αλλάξει την κατάσταση του sprint). Ακόμα μπορεί να πλοηγηθεί σε προηγούμενα sprints, αλλά και αυτά που είναι για το σύντομο μέλλον. Μπορεί να δημιουργήσει όσα sprints χρειάζεται. <br>
`Epic Stories Tasks Scenario`: Ο χρήστης μπoρεί να δημιουργήσει epic, stories και tasks. Να ελέγξει την πορεία τους και να πλοηγηθεί σε προηγούμενα. Να ελέγξει τις πληροφορίες τους όπως και να τις επεξεργαστεί αναλόγως. <br>
`Progress Scenario`: Ο χρήστης θα μπορεί να ελέγχει την πρόοδό του και την δουλειά του.
### 2.1.3 Security and Settings
`Settings Scenario`: Ο χρήστης θα μπορεί να τροποποιήσει όπως θέλει τις προσωπικές του ρυθμίσεις και χαρακτηριστικά, όπως π.χ. να αλλάξει το username του.<br>
`Security Scenario`: Οι κωδικοί του χρήστη και τα στοιχεία του θα αποθηκεύονται σε ασφαλή χώρο.<br>
`Verification Scenario`: Ο χρήστης θα πρέπει να κάνει activate το λογαριασμό του για να μπορεί να χρησιμοποιήσει σωστά την εφαρμογή .<br>
`Billing Method`: Σελίδα με δυνατότητες ρύθμισης των μεθόδων πληρωμής του χρήστη, αλλαγή σε κάποιο άλλο προσφερόμενο πακέτο, ή και διακοπή της συνδρομής.
### 2.1.5 Projects
`All Projects`: Μια λίστα που καθοδηγεί στα διαφορετικά projects του χρήστη.<br>
`New Project Button`: Οδηγείται στη σελίδα, για τη δημιουργία καινούργιου Project. Ο χρήστης καλείται να εισάγει όλα τα δεδομένα ενός Project, όπως τίτλος και περιγραφή, και να προσκαλέσει τους/της συνεργάτες του.<br>
`Project Page`: Παρέχονται όλες οι πληροφορίες για τα τρέχοντα Sprints, Epic αλλά και για προηγούμενα Sprints (past Sprints). 
1. **Sprints**:  Ενημερώνει το χρήστη για Sprints που "τρέχουν" εκείνη τη στιγμή. Sprints που πρόκειται να ξεκινήσουν στο άμεσο μέλλον, αλλά και στο πιο μακρινό. 
2. **Εpic**: Ενημερώνει το χρήστη για τα Epics που αποτελείται το τρεχόν Project. Δίνοντας του μερικές πληροφορίες γι'αυτό.
3. **Past Sprints**: Ενημερώνει το χρήστη για τα προηγούμενα Sprints.
4. **Create Issue Button**: Δημιουργία issue για το τρεχόν Project. Κατηγοριοποίηση του issue σε *Epic*, *Story*, *Task* και *Bug*. Επεξεργασία κατάλληλα τις πληροφορίες του Issue, αναλόγως της κατηγορίας του και ανάθεση developer που θα δουλέψει για το Issue.
### 2.1.6 Chat
`Contacts`: Σελίδα όπου ο χρήστης μπορεί να επικοινωνήσει με τις αποθηκευμένες του επαφές, και να δει και τις πληροφορίες τους.<br>
`Message`: Η αλληλεπίδραση του χρήστη με τους υπόλοιπους χρήστες της εφαρμογής.<br>
`Add Contact Button`: Δημιουργία καινούργιας επαφής στην εφαρμογή.<br>
`Search Bar`: Αναζήτηση στις επαφές για συγκεκριμένο χρήστη.<br>
`Groups`: Συνομιλίες μεταξύ πολλαπλών χρήστων.<br>
`Create Group`: Δημιουργία Group.<br>
`Join Group`: Συμμετοχή σε υπάρχον Group.
## <a name="head22"></a> 2.2 Κατηγορίες χρήστη και Χαρακτηριστικά
`Developer`: Η εφαρμογή απευθύνεται κυρίως σε προγραμματιστές. Ο λόγος για να τη χρησιμοποιήσουν θα είναι για να οργανωθούν και να ολοκληρώσουν κάποιο από τα Projects τους. Η εμπειρία τους με Scrum εφαρμογές μπορεί να κυμένεται σε διάφορες βαθμίδες. Ένας Developer καλείται να συνειδητοποιήσει πως λείτουργουν οι βασικές λειτουργίες του προγράμματος, χωρίς κάποιες ιδιαίτερες επεξηγήσεις πέρα από κάποιο documentation.  
`Scrum Master`: Ο Scrum Master είναι κι αυτός προγραμματιστής, αλλά έχει αναλάβει και την καθοδήγηση του Project. Θεωρείται ότι έχει μία πολύ καλή εμπειρία με τη χρήση Scrum εφαρμογών, αλλά δεν είναι και αναγκαίο.<br>
`Product Owner`: Ο Product owner χρειάζεται το τρεχόν προϊόν για να οργανώσει μία ομάδα προγραμματισμών για την υλοποίηση του Project. Θεωρείται ότι θα αγοράσει κάποιο από τα πακέτα λειτουργίας, για να το παρέχει και στους εργαζομένούς του. Η χρήση που περιμένουμε από τον Product Owner είναι από μηδαμινή εως και άπειρη, αφού μπορεί να ασχολείται και αυτός ως Developer στο Project. Οι γνώσεις του επάνω στην εφαρμογή θα είναι ανάλογες και με τη χρήση του. Στην περίπτωση όμως που δεν το χρησιμοποιεί δεν υπάρχει λόγος από την εφαρμογή να τον/την καθοδηγήσει. 
## <a name="head23"></a> 2.3 Περιβάλλον Λειτουργίας
Η εφαρμογή πρόκειται να υλοποιήθει ως site στον Παγκόσμιο Ιστό (World Wide Web). 
## <a name="head24"></a> 2.4 Περιορισμοί σχεδίασης και υλοποίησης
To front end της εφαρμογής θα είναι της τεχνολογίας SPA (Single Page Application). 
# <a name="head3"></a> 3. Εξωτερικές Απαιτήσεις Διεπαφής
## <a name="head32"></a> 3.2 Διεπαφή Λογισμικού
Η εφαρμογή υποστιρίζει ένα σύστημα REST API το οποίο υλοποίειται με τη χρήση Java σε συνδιασμό με την τεχνολογία Spock. Το front-end της εφαρμογής αποτελείται από javascript με τη χρήση του React Framework. Το build της εφαρμογής θα γίνεται με τη χρήση του Gradle. Η βάση δεδομένων στην οποία θα αποθηκεύονται τα απαραίτητα δεδομένα του προγράμματος θα είναι υλοποιημένη σε σύστημα είτε NoSQL ( πιθανώς mongodb) ή σε SQL ( MySQL ή PostgreSQL).
## <a name="head33"></a> 3.3 Διεπαφή Επικοινωνίας
Η εφαρμογή θα υποστηρίζει το πρωτόκολλο επικοινωνίας HTTPS.
# <a name="head4"></a> 4. Χαρακτηριστικά Συστήματος
## <a name="head41"></a> 4.1 Back-end
Υποσύστημα το οποίο θα υποστηρίζει ένα ενιαίο REST API για την εκτέλεση ενεργειών από τον χρήστη μέσω του CLI και της web εφαρμογής.
## <a name="head42"></a> 4.2 CLI (Command Line Interface)
Ο χρήστης θα έχει τη δυνατότητα να κάνει συγκεριμένες ενέργειες μέσω ενός CLI το οποίο θα είναι client του  REST  API  που  παρέχεται  από  το  back-end  υποσύστημα.
## <a name="head43"></a> 4.3 Front-end
Ένα web application για οργάνωση ομάδων scrum στο οποίο θα έχουν πρόσβαση οι χρήστες της πλατφόρμας μέσω ενός web browser. Θα είναι υλοποιημένο ως ένας δεύτερος client του REST API του back-end υποσυστήματος.
# <a name="head6"></a> Appendix A : Γλωσσάριο 
`Bug`: Μια μικρή διόρθωση της εφαρμογής που δεν χρειάζεται κάποιο ιδιαίτερο testing μετά.<br>
`Developer`: Προγραμματιστής<br>
`Epic`:  Είναι ένα μεγάλο μέρος μια δουλειάς το οποίο μπορεί να διασπαστεί σε σε μικρότερο αριθμό από stories. Tα epic συνήθως περιλαμβάνουν διάφορες ομάδες, σε διάφορα projects. Τα epics ολοκληρώνονται συνήθως σε πέρας αρκετων sprints.<br>
`Product Owner`: Είναι το άτομο που είναι υπεύθυνο για την συντήρηση των εκκρεμοτήτων του προϊόντος με το να εκπροσωπεί το συμφέρον των ενδιαφερόμενων και εξασφαλίζοντας την αξία της δουλειάς που κάνει η ομάδα ανάπτυξης.<br>
`Scrum`: Σκραμ είναι πλαίσιο επαναληπτικής και επαυξητικής ευέλικτης μεθοδολογίας ανάπτυξης λογισμικού για τη διαχείριση της ανάπτυξης ενός προϊόντος.<br>
`Scrum Master`: Ο Scrum Master είναι ο αρχήγος της ομάδα, και είναι υπεύθυνος για την σωστή υλοποιήση του project, καθοδηγώντας την ομάδα και τον Product owner, και ελέγχοντας ότι όλες οι πρακτικές του Scrum ακολουθούνται από την ομάδα.<br>
`Sprint`: Είναι ένα μικρό χρονικό διάστημα όπου η ομάδα scrum έχει αναλάβει να ολοκληρώσει κάποιον όγκο δουλειάς.<br>
`Story`: Είναι ένα μικρότερο πακέτο εργασιών.<br>
`Task`: Η τεχνική δουλειά που η ομάδα προγραμματιστών καλείται να υλοποιήσει για την ολοκλήρωση ενός αντικειμένου από το σωρό διεργασιών του προϊόντος. Συνήθως είναι μικρά, και δεν χρειάζεται παραπάνω εργασία από κάποιες ώρες εως και μέρα δουλειάς.
