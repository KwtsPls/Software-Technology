# <a name="head1"></a> 1. Εισαγωγή
## <a name="head11"></a> 1.1 Σκοπός  
Το παρακάτω κείμενο αποτελεί περιγραφή για τις προδιαγραφές απαιτήσεων λογισμικού. Θα αναλυθούν κυρίως σε πρωτόλειο στάδιο τα χαρακτηριστικά της εφαρμογής και οι τρόποι υλοποίησης.
## <a name="head12"></a> 1.2 Συμβάσεις Σύνταξης Έγγραφου
Το παρόν κείμενο έχει υλοποιηθεί σε MarkDown. Κάθε μέρος του κειμένου έχει χωριστεί σε κατηγορίες και υποκατηγορίες.
## <a name="head13"></a> 1.3 Προβλεπόμενο κοινό και προτάσεις ανάγνωσης 
Το κείμενο έχει δημιουργηθεί από φοιτητές και πρόκειται για μια εφαρμογή στα πλαίσια του μαθήματος Τεχνολογία Λογισμικού. Το κοινό στο οποίο απευθύνεται η εφαρμογή , πέρα από τους φοιτητές για τον εκπαιδευτικό σκοπό του μαθήματος, είναι εργαζόμενοι στον κλάδο του προγραμματισμού, οι οποίοι λειτουργούν στις ομάδες τους με τον κύκλο της μεθόδου ανάπτυξης λογισμικού Agile και πιο συγκεκριμένα με την τεχνική Scrum.
## <a name="head14"></a> 1.4 Περίληψη Εφαρμογής
Η εφαρμογή επιτρέπει την επικοινωνία μιας προγραμματιστικές ομάδας με την τεχνική Scrum. Δηλαδή, είναι μια εφαρμογή που χρησιμοποιείται για την καλύτερη οργάνωση ομάδας προγραμματιστών, με σκοπό να μπορέσουν να εκτελέσουν συγκεκριμένα μέρη ενός μεγαλύτερου προγράμματος. Για κάθε project, που δημιουργεί ένας χρήστης, μπορεί να προσκαλέσει τους υπόλοιπους προγραμματιστές και να δημιουργήσει κάποιο issue για το project. (#anchors-in-markdown)

Τα issues αποτελούνται από 3 μέρη, τα epics και τα stories και τα tasks/bugs. Το epic πρόκειται για ένα κεντρικό σκοπό της υλοποίησης της εφαρμογής που διαιρείται σε stories. Ενώ το story διαιρείται σε διάφορα tasks έτσι ώστε να υλοποιηθεί κάποια λειτουργία του epic. Ακόμη υπάρχουν και τα bugs όπου πρόκειται για την επίλυση προγραμματιστικών λαθών της εφαρμογής που χρήζουν διόρθωση. Τέλος, η ομάδα καλείται να φέρει εις πέρας κάποια tasks, μέσα σε κάποιο χρονικό διάστημα, γνωστό και ως sprint. Όταν τελειώσει ένα sprint, ξεκινάει το επόμενο, με καινούργια tasks.
## <a name="head15"></a> 1.5 Βιβλιογραφία 
Δεν έχει χρησιμοποιηθεί κάποια βιβλιογραφία ακόμη.
# <a name="head2"></a> 2. Γενική Περιγραφή 
## <a name="head21"></a> 2.1 Λειτουργικότητα Εφαρμογής
Θα χωρίσουμε την εφαρμογή σε 6 μέρη
### 2.1.1 Starting Page
`Login Page`: Η σελίδα όπου ο χρήστης θα συνδέεται στο λογαριασμό του.<br>
`Signup Page`: Η σελίδα όπου ο χρήστης θα δημιουργεί καινούργιο λογαριασμό στην εφαρμογή μας.<br>
`Payment Plan Page`: Η σελίδα όπου θα περιγράφονται τα πακέτα υπηρεσιών της εφαρμογής, επι πληρωμή ή δωρεάν.
### 2.1.2 Menu Bar
Μια κύρια μπάρα που οδηγεί σε κάθε κύρια λειτουργία της εφαρμογής.
### 2.1.3 Overview Page
`Welcome Window`: Ένα μικρό παραθυράκι που θα καλωσορίζει το χρήστη, με διαφορετικό τρόπο, και ίσως με κάποιο αστείο ή κάποια ενδιαφέρουσα πληροφορία.<br>
`Recent Projects`: Ένα μικρό παραθυράκι που θα εμφανίζονται όλα τα πιο πρόσφατα επεξεργασμένα projects.<br>
`Dashboard Window`: Το πρόγραμμα του χρήστη και η πρόοδος του στα project του.
### 2.1.4 Settings
`User's Profile`: Σελίδα με τα προσωπικά στοιχεία του χρήστη, επιλογές εξασφάλισης των προσωπικών του δεδομένων, και δυνατότητα αλλαγής τους.<br>
`Personalisation`: Σελίδα με δυνατότητες αλλαγής της διεπαφής του χρήστη, με βάση τις προσωπικές του προτιμήσεις.<br>
`Account-Security`: Σελίδα με δυνατότηες αλλαγής του κωδικού πρόσβασης και ενεργοποίησης του λογαριασμού.<br>
`Billing Method`: Σελίδα με δυνατότητες ρύθμισης των μεθόδων πληρωμής του χρήστη, αλλαγή σε κάποιο άλλο προσφερόμενο πακέτο, ή και διακοπή της συνδρομής.
### 2.1.5 Projects
`All Projects`: Μια λίστα που καθοδηγεί στα διαφορετικά projects του χρήστη.<br>
`New Project Button`: Οδηγείται στη σελίδα, για τη δημιουργία καινούργιου Project. Ο χρήστης καλείται να εισάγει όλα τα δεδομένα ενός Project, όπως τίτλος και περιγραφή, και να προσκαλέσει τους/της συνεργάτες του.<br>
`Project Page`: Παρέχονται όλες οι πληροφορίες για τα τρέχοντα Sprints, Epic αλλά και για προηγούμενα Sprints (past Sprints). 
1. **Sprints**:  Ενημερώνει το χρήστη για Sprints που "τρέχουν" εκείνη τη στιγμή. Sprints που πρόκειται να ξεκινήσουν στο άμεσο μέλλον, αλλά και στο πιο μακρινό. 
2. **Εpic**: Ενημερώνει το χρήστη για τα Epics που αποτελείται το τρεχόν Project. Δίνοντας του μερικές πληροφορίες γι'αυτό.
3. **Past Sprints**: Ενημερώνει το χρήστη για τα προηγούμενα Sprints.
4. **Create Issue Button**: Δημιουργία issue για το τρεχόν Project. Κατηγοριοποίηση του issue σε *Epic*, *Story*, *Task* και *Bug*. Επεξεργασία κατάλληλα τις πληροφορίες του Issue, αναλόγως της κατηγορίας του και ανάθεση developer που θα δουλέψει για το Issue.
### 2.1.6 Chat
`Contacts`: Σελίδα όπου ο χρήστης μπορεί να επικοινωνήσει με τις αποθηκευμένες του επαφές, και να δει και τις πληροφορίες τους.<br>
`Message`: Η αλληλεπίδραση του χρήστη με τους υπόλοιπους χρήστες της εφαρμογής.<br>
`Add Contact Button`: Δημιουργία καινούργιας επαφής στην εφαρμογή.<br>
`Search Bar`: Αναζήτηση στις επαφές για συγκεκριμένο χρήστη.<br>
`Groups`: Συνομιλίες μεταξύ πολλαπλών χρήστων.<br>
`Create Group`: Δημιουργία Group.<br>
`Join Group`: Συμμετοχή σε υπάρχον Group.
## <a name="head22"></a> 2.2 Κατηγορίες χρήστη και Χαρακτηριστικά
`Developer`: Η εφαρμογή απευθύνεται κυρίως σε προγραμματιστές. Ο λόγος για να τη χρησιμοποιήσουν θα είναι για να οργανωθούν και να ολοκληρώσουν κάποιο από τα Projects τους. Η εμπειρία τους με Scrum εφαρμογές μπορεί να κυμένεται σε διάφορες βαθμίδες. Ένας Developer καλείται να συνειδητοποιήσει πως λείτουργουν οι βασικές λειτουργίες του προγράμματος, χωρίς κάποιες ιδιαίτερες επεξηγήσεις πέρα από κάποιο documentation.  
`Scrum Master`: Ο Scrum Master είναι κι αυτός προγραμματιστής, αλλά έχει αναλάβει και την καθοδήγηση του Project. Θεωρείται ότι έχει μία πολύ καλή εμπειρία με τη χρήση Scrum εφαρμογών, αλλά δεν είναι και αναγκαίο.<br>
`Product Owner`: Ο Product owner χρειάζεται το τρεχόν προϊόν για να οργανώσει μία ομάδα προγραμματισμών για την υλοποίηση του Project. Θεωρείται ότι θα αγοράσει κάποιο από τα πακέτα λειτουργίας, για να το παρέχει και στους εργαζομένούς του. Η χρήση που περιμένουμε από τον Product Owner είναι από μηδαμινή εως και άπειρη, αφού μπορεί να ασχολείται και αυτός ως Developer στο Project. Οι γνώσεις του επάνω στην εφαρμογή θα είναι ανάλογες και με τη χρήση του. Στην περίπτωση όμως που δεν το χρησιμοποιεί δεν υπάρχει λόγος από την εφαρμογή να τον/την καθοδηγήσει. 
## <a name="head23"></a> 2.3 Περιβάλλον Λειτουργίας
Η εφαρμογή πρόκειται να υλοποιήθει ως site στον Παγκόσμιο Ιστό (World Wide Web). 
## <a name="head24"></a> 2.4 Περιορισμοί σχεδίασης και υλοποίησης
To front end της εφαρμογής θα είναι της τεχνολογίας SPA (Single Page Application).
